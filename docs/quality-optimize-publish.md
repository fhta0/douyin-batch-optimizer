# 质量分去优化自动发布功能开发文档

## 1. 功能目标

在抖店商品管理页自动处理 `质量分` 列中可点击的 `去优化` 项，进入商品编辑页后，按页面可用能力自动执行图片优化动作，并在满足条件时点击 `发布商品`，最后关闭编辑页返回商品列表。

目标页面：

- 商品管理页：`https://fxg.jinritemai.com/ffa/g/list`
- 商品编辑页：`https://fxg.jinritemai.com/ffa/g/create?...&entrance=edit&diagnose=1`

## 2. 本次功能边界

本功能只处理以下链路：

1. 在商品管理页找到 `质量分` 列中的 `去优化`
2. 点击 `去优化`，新开商品编辑页
3. 在编辑页处理：
   - `从1:1主图智能裁剪`
   - `一键生成`
   - `发布商品`
4. 完成后关闭当前编辑页，回到商品管理页继续下一条

## 3. 页面现状观察

通过真实页面 MCP 验证，已确认：

1. `去优化` 在商品管理页中是可点击按钮，真实节点形态类似：
   - `button.ecom-g-btn.ecom-g-btn-link.ecom-g-btn-sm`
   - 文案：`去优化`

2. 点击 `去优化` 后会新开商品编辑页，而不是当前页跳转。

3. 在编辑页中：
   - `从1:1主图智能裁剪` 确实可点击
   - 点击后按钮文案会变成 `取消智能裁剪`
   - `一键生成` 并不是所有商品页都会出现，因此必须按“可选能力”处理，不能写死为必有

4. `发布商品` 按钮位于页面底部固定操作区，可通过明确文案识别。

## 4. 交互流程

### 4.1 列表页流程

1. 打开商品管理页
2. 在当前页表格中查找 `去优化`
3. 找到后点击第一个可见且可交互的 `去优化`
4. 监听页面是否新开标签页
5. 切换到新标签页执行编辑流程

### 4.2 编辑页流程

1. 等待编辑页基础结构加载完成
   - 页面标题区域出现
   - `图文信息` 区域已渲染
   - 页面底部 `发布商品` 可识别

2. 处理 `从1:1主图智能裁剪`
   - 如果按钮存在且文案是 `从1:1主图智能裁剪`，则点击
   - 点击后等待按钮文案变为 `取消智能裁剪`，或页面图片区块出现裁剪结果变化
   - 如果按钮已是 `取消智能裁剪`，视为已处理，直接跳过

3. 处理 `一键生成`
   - 如果按钮存在且可点击，则点击
   - 点击后等待生成完成
   - 完成标志建议从以下信号择一或组合：
     - 按钮进入不可点击态后再次恢复
     - 页面出现“生成成功”类提示
     - 生成中的 loading 消失
     - 相关图片区块内容发生变化
   - 如果页面不存在 `一键生成`，直接跳过，不视为失败

4. 处理 `发布商品`
   - 仅在前置步骤完成后点击 `发布商品`
   - 点击后等待发布成功信号
   - 发布完成后关闭当前标签页

5. 返回商品管理页继续下一条

## 5. 顶层状态

建议增加独立于”属性批量优化”的第二套任务状态：

- `idle` — 空闲
- `scanning_list` — 扫描列表页
- `opening_optimize_page` — 打开编辑页
- `processing_edit_page` — 处理编辑页
- `publishing` — 发布中
- `closing_tab` — 关闭标签页
- `completed` — 完成
- `error` — 错误

编辑页内部的子状态（如等待加载、裁剪中、生成中等）由实现内部管理，不在此定义。

## 6. 页面识别策略

### 6.1 商品管理页识别

满足以下条件之一即可：

- URL 包含 `/ffa/g/list`
- 页面出现 `商品管理`
- 表格列中出现 `质量分`

### 6.2 编辑页识别

满足以下条件之一即可：

- URL 包含 `/ffa/g/create`
- 页面出现 `图文信息`
- 页面底部出现 `发布商品`

## 7. 选择器策略

不要只依赖 class 名，优先使用“文案 + 角色 + 局部作用域”。

### 7.1 列表页

优先查找：

- 表格区域内，文案为 `去优化` 的 `button`

建议限制在商品表格容器内查找，避免命中侧边栏或其他同名按钮。

### 7.2 编辑页

#### 从1:1主图智能裁剪

优先查找：

- 文案为 `从1:1主图智能裁剪` 的 `button`
- 或文案为 `取消智能裁剪` 的 `button`

#### 一键生成

优先查找：

- 文案为 `一键生成` 的 `button`

#### 发布商品

优先查找：

- 文案为 `发布商品` 的底部主按钮

## 8. 等待策略

每一步采用"优先信号 + 超时兜底"模式，不使用固定 sleep。

### 8.1 打开编辑页后

| 优先信号 | 超时兜底 |
|---------|---------|
| `图文信息` 区域出现 + `发布商品` 按钮可识别 + 页面无 loading 状态 | 30 秒超时，关闭并跳过 |

### 8.2 点击智能裁剪后

| 优先信号 | 超时兜底 |
|---------|---------|
| 按钮文案变为 `取消智能裁剪` | 10 秒超时，记录失败继续 |

### 8.3 点击一键生成后

| 优先信号 | 超时兜底 |
|---------|---------|
| 成功 toast 出现 或 按钮恢复可点击 | 30 秒超时，记录失败继续 |

### 8.4 点击发布商品后

| 优先信号 | 超时兜底 |
|---------|---------|
| 成功 toast 出现 | 30 秒超时，保留页面并暂停 |

## 9. 异常处理

### 9.1 去优化不存在

- 当前页没有可处理 `去优化`，则本页结束
- 如果后续没有翻页策略，可直接结束任务
- 如果后续要支持翻页，则翻到下一页继续查找

### 9.2 智能裁剪按钮不存在

- 直接跳过
- 记录日志：`当前商品无主图智能裁剪入口，已跳过`

### 9.3 一键生成按钮不存在

- 直接跳过
- 记录日志：`当前商品无一键生成入口，已跳过`

### 9.4 发布按钮不可点击

- 记录失败原因
- 保留当前标签页，避免丢失现场
- 日志中写明：`发布商品按钮不可点击，已保留当前页面`

### 9.5 发布后长时间无反馈

- 进入超时分支
- 建议不自动关闭页面
- 记录日志：`发布结果等待超时，请人工确认`

## 13. 当前结论

这个功能是可做的，但实现时必须按“可选能力”设计，而不是强依赖每个页面都有相同按钮。

当前已确认的事实：

- `去优化` 可稳定打开编辑页
- `从1:1主图智能裁剪` 在部分页面可稳定触发
- `一键生成` 不是每个页面都有
- `发布商品` 是最终高风险动作，必须有明确成功判定和异常保留策略

## 14. 下一步实现口径

建议按以下最终口径开发：

1. 列表页找到 `去优化`
2. 打开编辑页
3. 有 `从1:1主图智能裁剪` 就点
4. 有 `一键生成` 就点并等待完成，没有就跳过
5. 点击 `发布商品`
6. 发布成功后关闭当前页
7. 回到列表页继续下一项

这也是当前最稳、最接近真实页面差异的实现方案。

## 15. 与属性优化功能并存的方案

新功能不建议做成“属性优化”和“图文优化”二选一，而是建议通过设置中心的勾选项来决定本次运行哪些能力。

推荐的并存方式：

- `属性优化（批量）`
- `图文优化（包括主图3:4、主图视频）`

### 15.1 推荐执行规则

1. 如果勾选了 `属性优化（批量）`
   - 执行现有的批量属性优化流程

2. 如果勾选了 `图文优化（包括主图3:4、主图视频）`
   - 在商品管理页扫描 `去优化`
   - 打开商品编辑页
   - 有 `从1:1主图智能裁剪` 就执行
   - 有 `一键生成` 就执行
   - 图文处理完成后自动点击 `发布商品`
   - 发布成功后关闭当前页

3. 如果两个都勾选
   - 默认执行顺序建议为：
     - 先 `属性优化（批量）`
     - 再 `图文优化（包括主图3:4、主图视频）`

### 15.2 图文优化自动发布的最终规则

这是整个功能最关键的防误操作约束：

**规则 1：无优化则不发布**
- 如果当前商品页既没有 `从1:1主图智能裁剪`，也没有 `一键生成`
- 则直接关闭编辑页并跳过，不发布

**规则 2：有优化才允许发布**
- 如果至少执行了以下任一项：`从1:1主图智能裁剪` 或 `一键生成`
- 则允许自动点击 `发布商品`

**规则 3：失败保留现场**
- 如果 `发布商品` 失败或等待结果超时
- 保留当前编辑页，暂停任务，等待人工确认

### 15.4 实现建议

建议后续实现时拆成两个 runner，由总调度器统一编排：

- `attributeOptimizationRunner`
- `qualityOptimizeRunner`

总调度器按设置项决定：

1. 本次是否运行属性优化
2. 本次是否运行图文优化

这样能保证：

- 设置中心足够简洁

## 17. 第一版必处理异常规则

以下 5 条是第一版必须处理的异常，其他边界情况放入后续增强。

### 17.1 发布规则：无优化操作则跳过发布

**规则**: 如果当前商品页既没有执行智能裁剪，也没有执行一键生成，则直接关闭编辑页并跳过，不发布。

**理由**: 避免无意义的发布操作，符合 15.2 节定义的发布规则。

**处理方式**:
```javascript
state.cropExecuted = false;
state.generateExecuted = false;
// ... 执行裁剪和生成 ...
if (!state.cropExecuted && !state.generateExecuted) {
  log('[发布] 无实际优化操作执行，跳过发布');
  await closeEditorTab();
  return;
}
```

### 17.2 发布失败或超时：保留页面并暂停

**规则**: 发布失败或等待发布结果超时时，保留当前编辑页，暂停任务，等待人工确认。

**理由**: 发布是高风险操作，失败时不应自动跳过或关闭页面，需要人工介入。

**处理方式**:
- 不自动关闭编辑页
- 状态置为暂停
- 日志提示用户人工确认

### 17.3 操作超时规则

**规则**: 所有关键操作均设置超时，超时后记录日志并采取对应措施。

**第一版超时配置**:
| 操作 | 默认超时 | 超时后行为 |
|------|---------|-----------|
| 编辑页加载 | 30秒 | 关闭编辑页，跳过当前商品 |
| 智能裁剪 | 10秒 | 记录失败，继续后续流程 |
| 一键生成 | 30秒 | 记录失败，继续后续流程 |
| 发布结果 | 30秒 | 保留页面，暂停任务 |

### 17.4 按钮状态检测

**规则**: 点击按钮后等待状态变化确认操作生效，而非固定 sleep 等待。

**检测方式**:
- 智能裁剪：等待按钮文案变为"取消智能裁剪"
- 一键生成：等待成功 toast 或按钮恢复可点击
- 发布商品：等待成功 toast 或页面关闭

### 17.5 确认弹窗处理

**规则**: 点击发布商品后如果出现确认弹窗，自动点击确认按钮。

**处理方式**: 检测 `[class*="modal"]` 等弹窗元素，查找包含"确认"文案的按钮并点击。

以下功能在后续版本考虑实现，第一版不做：

### 18.1 Tab 监听与跨标签调度

当前架构以 content.js 为主，不依赖 background/service worker。后续如需支持：
- 监听标签页关闭事件
- 跨标签页状态同步
- 多标签页并行处理

需要新增 background 调度层，架构改动较大，放后续评估。

### 18.2 发布成功二次验证

发布成功后返回列表页验证商品状态变化。第一版采用简化策略：**信任发布成功信号**。

理由：
- 增加复杂度和循环检测风险
- 平台侧问题脚本侧难以完全解决
- 失败时已有保留页面机制兜底

### 18.3 同一商品重复处理检测

通过商品 ID 去重，避免列表页未刷新时重复处理同一商品。第一版不做，依赖用户手动刷新列表页。

### 18.4 用户手动干预检测

监听用户意外点击或标签页关闭，自动暂停任务。第一版不做，依赖用户主动点击"停止"按钮。

### 18.5 翻页处理

当前页无"去优化"时自动翻页继续处理。第一版不支持翻页，当前页处理完毕即结束任务。
